An Gramadóir:
Porting to other languages

Introduction

I would be excited to help port this software for use with languages other than Irish. Getting something simple up and running should be easy, even if your language lacks some of the basic tools for doing natural language processing. If, however, you already have things like a large word list or a part-of-speech tagger, then you should be able to begin implementing sophisticated grammatical rules immediately.

Caveat Emptor: If you've read the Details page, you know that An Gramadóir is a rule-based grammar checker that uses pattern matching to find errors. This is quite satisfactory for Irish where many of the common errors made in writing involve misuse of the initial mutations which are determined almost entirely by local context (usually, just the preceding word). It is much harder to find errors in languages like English using the same scheme, and harder still in languages having free word order.

Things to do

1. Build focail.bs. This is the database of recognized words. If you have some grammatical rules that don't require part of speech information, then in fact this step is optional. You can either construct focail.bs from a plain text file containing one word per line (which can be generated easily if your language has ispell support):

cat wordlist | sed 's/.*/& 1/' | cabhair > focail.bs

or get rid of the first rule in rialacha.in to turn off spellchecking if you have no word list at all. An intermediate possibility would be if, for instance, you want to check for erroneous constructions like the English "He done went fishing", or "He done ate his dinner" (sadly common in my part of the world). You could add a rule to rialacha.in resembling:

<N>ANYTHING</N> done <VP>ANYTHING</VP>:MESSAGE

and populate focail.bs with verbs in the past tense. It all depends on what kind of checks you need to do for your language. You can also provide a "replacement" file which contains misspelled (or, as in the case of Irish, pre-standard or dialect) words and suggested replacements. See eile.bs in the distribution.

2. Rewrite "byte_to_markup" in cuardach.c. You can choose the grammatical codes that go into focail.bs as you see fit; just decode them correctly in this one function.

3. Tweak the details. If your language has unusual conventions for ends of sentences, you may need to play with the abairti module. Or if you need characters outside of Latin-1, the di-utf function will need to be generalized. The buffers in cuardach.c for storing Irish words are 32 bytes long (longest word: ngearr-chlóscríobhneoireachta, 28 bytes) and may need to be increased. There may be other minor things as well.

4. Write aonchiall.in. These are the rules for disambiguation. In Irish, and I would imagine in many other languages, one can get away without too many rules here. See the description of this file on the Contributing page.

5. Write rialacha.in and eisceacht.in. These are the grammatical rules and exceptions. Again see the Contributing page for information on how to create these files.

6. Localize message strings. Technically this is part of porting the user interface, not porting the grammar checker. I'm guessing however that you'll want to see the output in the same language as the documents that are being checked. I've kept these strings together at the top of gr.in. There are also a couple of rarely seen error messages in cuardach.c. I'll move to using gettext in the future to streamline this.
